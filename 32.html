Spring Cloud Alibaba 大型微服务项目实战

在《微服务落地一站式解决方案——Spring Cloud》章节中介绍了常用的微服务架构应包含哪些组件，以及Spring Cloud Alibaba套件中对应的落地方案。Spring Cloud Alibaba套件中服务容错方面的核心组件就是Sentinel。本章节就对Sentinel展开讲解，主要包括如下内容：

  • 服务容错模块的重要性和功能
  • 服务容错落地方案：限流与熔断
  • Sentinel介绍及控制台安装

为什么要引入服务容错？

随着微服务架构的普及和项目的实际落地，服务容错的概念也渐渐流行开来。开发者们也更加关注服务的稳定性，像Hystrix、Sentinel这些用于保护服务稳定运行的组件也开始涌现和发展，在保证服务的稳定性和系统的高可用中扮演着重要的角色。

高可用（high availability，即 HA），其主要目的是为了保障“业务的连续性”。即在用户眼里，系统永远是正常（或者说基本正常）对外提供服务的，高可用是架构设计和系统搭建时必须要考虑的一点。而谈到高可用，很多开发者脑海中第一个出现的技术就是“搭建集群”，通过搭建集群能够有效避免由于单点故障导致的系统不可用。集群化是保障系统“高可用”的基本操作，不过，只是搭建集群还不够，还需要对系统做更多的优化工作，比如优化业务代码、引入缓存、引入异步操作的逻辑、优化 SQL 语句、增加服务器的物理配置等等。

上述的系统优化手段已经极大的提升了系统的健壮性。但是在微服务环境下，受制于硬件、网络、机器性能、算法、程序各方面影响，运行异常的情况也在显著提升，如果不做好异常保护，微服务架构就像空中楼阁一样随时可能会崩溃。现代微服务架构都是分布式的，由非常多的服务组成。不同服务之间相互调用，组成复杂的调用链路。

即使是单个服务出现了不可用的情形，在链路调用中会产生放大的效果。复杂链路上的某一环不稳定， 就可能会层层级联，最终导致整个链路都不可用。

这就是微服务系统高可用性的杀手——“服务雪崩”。“雪崩”一词指的是山地积雪由于底部溶解等原因造成的突然大块塌落的现象，具有很强的破坏力。服务雪崩效应是一种因“服务提供者的不可用”（原因）导致“服务调用者不可用”（结果），并产生级联效应将“不可用”逐渐放大的现象，最终导致整个系统处于不可用。

服务雪崩是一种笼统的描述，生产环境中会出现如下这些不稳定的状况：

  1. 活动期间导致瞬时流量过大，超出服务器物理机的承受范围，进而导致无法正常处理请求，用户只能看到一张空白的页面却无法操作。
  2. 热点数据击穿缓存，导致DB层被打垮，用户无法获取正常的数据也无法进行正常的操作。
  3. 调用端服务被不稳定的下游服务拖垮，导致某条调用链路卡死，用户会看到某个功能一直处于不可用的情况。

为了避免上述的这些问题，同时给微服务系统加上一层防护。保障系统的稳定性和高可用性，这就是引入服务容错的原因。

服务容错落地方案：流量控制与降级熔断

讲解完为什么要引入服务容错后，接下来笔者就要讲解服务容错具体的落地方案。

以服务雪崩效应为例，这种对系统造成破坏性伤害的状况有两个主要因素：

  • 瞬时流量太大，超出了系统的负载。
  • 部分服务响应不及时，进而逐步拖垮了上游服务。

因此服务容错的落地方案就要针对这两个问题进行处理。首先进行流量控制，可以采用限流的方案控制请求数量，让请求流量有序的进入应用，并保证流量在一个可控的范围内。其次是针对服务内部响应慢的问题，引入断路器模式，当调用链路中某个服务出现不稳定的情况（比如调用一直超时或者异常比例较高），则对这个服务的调用进行限制，并让请求快速失败，避免影响到系统中其它的服务。

有了解决问题的方案，就可以讲一讲具体落地实现了，也就是服务容错章节的主角——Sentinel。它是面向分布式、多语言异构化服务架构的流量治理组件，主要以流量为切入点，从流量路由、流量控制、流量整形、熔断降级、系统自适应过载保护、热点流量防护等多个维度来帮助开发者保障微服务的稳定性。

流量控制

请求流量具有随机性且很难准确预测，前一秒可能还风平浪静，后一秒可能就出现流量洪峰了，然而系统的容量总是有限的，如果突然而来的流量超过了系统的承受能力，就可能会导致请求处理不过来，堆积的请求处理缓慢，物理服务器的CPU和内存使用率持续升高，最后导致系统崩溃。因此就需要针对这种突发的流量来进行限制，在尽可能处理请求的同时来保障服务不被打垮，这就是流量控制。

Sentinel组件可以根据需要把随机的请求调整成合适的形状，如下图所示。

Sentinel组件中的流量控制有以下几个角度:

  • 资源的调用关系，例如资源的调用链路，资源和资源之间的关系；
  • 运行指标，例如QPS、线程池、系统负载等；
  • 控制的效果，例如直接限流、冷启动、排队等。

这些内容，笔者会在后续章节中进行详细的介绍和演示。

熔断降级

微服务架构都是分布式的，通常会有多个服务实例。在进行具体的功能实现时，一个服务常常会调用别的下游服务。然而，这个被依赖服务的稳定性是不能完全保证的。如果依赖的服务出现了不稳定的情况，则上游服务对该服务的请求响应时间变长，异常调用的数量积累起来的话可能就会拖垮上游服务，也就是人们常说的“一颗老鼠屎坏了一锅粥”。因此就需要对不稳定的依赖服务进行熔断降级，暂时切断不稳定调用，避免局部不稳定因素导致整体的雪崩。

所谓降级，是指当服务调用发生了响应超时、服务异常等情况时，跳出当前的执行方法，在服务内部执行一段“降级逻辑”，在具体的编码中就是再多写一个兜底方法，在这个方法里可以什么都不做，也可以直接返回提示信息或者做其它处理。

所谓熔断，则是直接拒绝访问了。一般是根据请求失败率或者请求响应时间做熔断。熔断好比家里的漏电保护器，当线路过热就会跳闸，以免烧坏线路。在Sentinel组件中，可以根据实际情况在控制台进行熔断规则的配置，可选的维度有异常数、异常比例和平均响应时间。

提到微服务技术栈中的熔断组件，其实开发者们对于Netflix套件中的Hystrix更加熟悉，毕竟它也是红极一时的服务容错方案。不过，随着Netflix套件进入维护期，Hystrix基本上已经停止更新了。之后，Sentinel组件、Resilience4j组件逐渐成为Hystrix在服务容错领域的替代方案。

Sentinel简介及控制台安装

阿里的流量防卫兵——Sentinel

Sentinel阿里巴巴技术团队于2018年开源的一款中间件产品，官方定义为：

面向分布式、多语言异构化服务架构的流量治理组件。

Sentinel官方社区页面如下图所示。

Sentinel被称为分布式系统的流量防卫兵，是阿里开源的流控框架，从服务限流、降级、熔断等维度来保护服务和系统稳定。历经阿里巴巴近10年大流量的考验，非常值得信赖。Sentinel同时提供了简介易用的控制台，可以看到接入应用的资源数据，并且可以在控制台设置一些规则来保护应用。它比Hystrix支持的范围广，如Spring Cloud、Dubbo、gRPC都可以很方便的整合。集成非常简单，只需要少量的配置和代码就能够轻松集成到项目中，也很容易完成一些定制化的逻辑。

2012 年，Sentinel诞生于阿里巴巴集团内部，主要功能为入口流量控制；

2013至2017 年，Sentinel在阿里巴巴集团内部迅速发展，成为基础技术模块，覆盖了所有的核心场景。Sentinel也因此积累了大量的流量归整场景以及生产实践。

2018年7月，阿里巴巴宣布限流降级框架组件Sentinel正式开源，并持续演进。国内的微服务开源领域迎来了一位新成员——Alibaba Sentinel。

2019年4月，Sentinel 贡献的spring-cloud-circuitbreaker-sentinel 模块正式被Spring Cloud社区合并至 Spring Cloud Circuit Breaker，成为Spring Cloud官方的主流推荐选择之一。

2022 年，Sentinel品牌升级为流量治理，领域涵盖流量路由/调度、流量染色、流控降级、过载保护/实例摘除等；同时社区将流量治理相关标准抽出到OpenSergo标准中，Sentinel作为流量治理标准实现。

以上就是Sentinel的发展历程。

Sentinel同时具备如下优良的特性：

  • 丰富的应用场景

    Sentinel 承接了阿里巴巴近10年的双十一大促流量的核心场景，例如秒杀（即突发流量控制在系统容量可以承受的范围）、消息削峰填谷、集群流量控制、实时熔断下游不可用应用等。

  • 完备的实时监控

    Sentinel同时提供实时的监控功能。开发者们可以在控制台中看到接入应用的单台机器秒级数据，甚至500台以下规模的集群的汇总运行情况。

  • 广泛的开源生态

    Sentinel 提供开箱即用的与其它开源框架的整合模块，例如与Spring Cloud、Dubbo、gRPC的整合，只需要引入相应的依赖并进行简单的配置即可快速地接入 Sentinel。

  • 完善的SPI扩展点

    Sentinel 提供简单易用、完善的 SPI 扩展接口。开发者们可以通过实现扩展接口来快速地定制逻辑。例如定制规则管理、适配动态数据源等。

下载与启动Sentinel控制台

**Sentinel分为两个部分，包括Sentinel控制台和Sentinel客户端。**读者切不可把这两块内容弄混了。

Sentinel客户端需要集成在Spring Boot微服务实例中，用于接收来自Dashboard配置的各种规则，并通过Spring MVC Interceptor拦截器技术实现应用限流、熔断保护。

Sentinel提供了一个轻量级的开源控制台，它提供机器发现以及健康情况管理、监控（单机和集群），规则管理和推送的功能。Sentinel客户端并不依赖Sentinel控制台，但是结合控制台可以取得最好的效果，有一个可视化的页面更方便也更直观。
