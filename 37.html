新增公共模块

随着模块的增加，肯定会有一些类是重复出现在各个模块中。这些公共的类大部分模块都会使用，但是在开发时不可能在每个模块中都保留这些重复的内容。在实际开发中，通常的做法就是新建一个公共模块，然后其它的业务模块将其作为依赖项引入进去。

公共模块主要用于存放一些公共的类，比如分页工具类、全局响应结果类、工具类等等。

在工程中新建一个newbee-mall-cloud-common模块，并在主pom.xml文件中增加该模块的配置，代码如下：

xml
复制代码
<modules>
  <module>newbee-mall-cloud-user-service</module>
  <module>newbee-mall-cloud-gateway-admin</module>
  <!-- 新增common模块 -->
  <module>newbee-mall-cloud-common</module>
</modules>

子节点newbee-mall-cloud-common模块的pom.xml文件如下：

xml

用户微服务模块改造

前文中实战编码结果的newbee-mall-cloud-step02项目中，用户微服务是一个单独的模块。而最终的微服务架构实战项目中，用户微服务模块的结构如下所示：

java
复制代码
newbee-mall-cloud-user-service  // 用户微服务
	├── newbee-mall-cloud-user-api // 存放用户模块中暴露出去的用于远程调用的FeignClient类
	└── newbee-mall-cloud-user-web // 用户模块API的代码及逻辑

分为user-web和user-api两个模块，各自的作用也都标注出来了。

这个做法其实很好理解，微服务架构的项目中是存在不同微服务实例的远程通信的，在当前的实战项目中是通过OpenFeign来实现，因此笔者就对用户微服务做了一次结构整理，单独创建了一个newbee-mall-cloud-user-api模块用户存放用户模块中暴露出去的用于远程调用的FeignClient类。这样，如果有哪个微服务需要调用用户微服务中的一些接口，直接将newbee-mall-cloud-user-api添加到其依赖项中即可使用FeignClient类中的方法来调用了。不止是用户微服务，其它业务微服务的模块结构亦是如此。

另外，newbee-mall-cloud项目中存放FeignClient类的模块，其命名方式都是xxx-api。这个命名方式是参考了Dubbo项目中的命名，其实也可以定义为newbee-mall-cloud-user-feign、newbee-mall-cloud-user-feignclient等等。

OpenFeign编码暴露远程接口

接下来，在newbee-mall-cloud-user-api模块中新增需要暴露出去的接口FeignClient。

新建ltd.user.cloud.newbee.openfeign包，之后在该包下新增NewBeeCloudAdminUserServiceFeign类，用于创建对管理员相关接口的feign调用。

NewBeeCloudAdminUserServiceFeign类代码如下：

java
复制代码
package ltd.user.cloud.newbee.openfeign;

import org.springframework.cloud.openfeign.FeignClient;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;

@FeignClient(value = "newbee-mall-cloud-user-service", path = "/users")
public interface NewBeeCloudAdminUserServiceFeign {

    @GetMapping(value = "/admin/{token}")
    String getAdminUserByToken(@PathVariable(value = "token") String token);
}

这个方法的作用就是根据token值来获取管理员数据，作鉴权使用。调用的接口是newbee-mall-cloud-user-web模块下NewBeeMallCloudAdminUserController类中的getAdminUserByToken()方法。这样的话，如果其它服务需要与用户微服务进行远程通信获取管理员数据，就可以引入newbee-mall-cloud-user-api模块作为依赖，并且直接调用getAdminUserByToken()方法即可。

另外，上述代码只是测试写法，后续会根据实际的接口设计进行修改。这里主要是为了讲解newbee-mall-cloud-user-service模块的目录结构设计，以及该模块下的两个子模块newbee-mall-cloud-user-api和newbee-mall-cloud-user-web的作用和目录规范。

远程调用feign包应该如何设置

这是一个拓展思考问题，微服务项目中的远程调用FeignClent该写在哪里？远程调用feign包该建在哪个模块里？当然，这里只是笔者根据过往的开发经验来总结的，并不是标准答案，只是作为一个引申的知识点讲解给各位读者。

比如，订单微服务需要调用用户微服务下的/getUserDetail/{userId}接口来获取用户信息。这个接口是已经在用户微服务实现了且功能正常，如果想要通过OpenFeign远程调用，就需要实现一个FeignClient类，然后订单微服务里引入这个FeignClient类并开启OpenFeign调用就完成了。这个FeignClient类可以在订单微服务工程中编写，也可以在用户微服务工程中编写，甚至有些团队会单独再创建一个feign模块用于存放所有的FeignClient。毕竟，只要引入OpenFeign的依赖Jar包就可以编写FeignClient类，不需要与业务代码耦合。那么，也就引出了一个问题：FeignClient是写在调用端（比如案例中的订单微服务）、还是写在被调用端（比如案例中的用户微服务）、还是写在单独创建的feign模块里呢？

newbee-mall-cloud项目中的目录设计是写在被调用端，各个微服务下都单独创建了对应的xxx-api模块用于存放FeignClient类，如果有其它微服务需要远程调用，直接引入对应的xxx-api依赖，然后就能够发起远程调用了。如此设计的原因如下：

正常企业开发模式下，服务的细节是不对外公开的，如果A业务组想要B业务组提供一个远程调用接口，直接让B业务组编码即可，因为A业务组的开发人员不熟悉B业务组的代码，这种情况下肯定让B业务组来提供更加正常、合理。不仅如此，这种方式也更加灵活和安全，B业务组想要暴露出哪些业务接口出去，自行实现并暴露出去即可，不想暴露的就不用写在feign包里了。比如用户微服务下有100个接口，需要暴露的接口有3个，开发团队只需要在FeignClient类中暴露这3个接口即可。

如果写在A业务组的微服务里，那么A业务组的开发人员可能还需要查看被B业务组负责的微服务里有哪些接口，然后再去写FeignClient类，这种做法也是不合理的，增加了上游微服务开发团队的工作量，而且如果每个开发团队都自行实现FeignClient的话，代码又是重复的。所以，由被调用方来做这个工作会更好一点。

当然，如果所有的FeignClient类都统一放在一个单独的模块下，也是一种实现方式。比如newbee-mall-cloud项目中，可以新建一个newbee-mall-cloud-feign模块，当前所有xxx-api模块下的FeignClient类都可以统一放在newbee-mall-cloud-feign模块中，在需要远程调用时直接引入newbee-mall-cloud-feign模块即可。

总结

本章节中牵涉的业务编码不多，也不复杂，主要讲解的知识点是公共模块的创建、项目的目录规范和服务通信OpenFeign的引入，后续实战章节中的目录结构基本都是按照这个规范来做的。希望各位读者能够根据笔者提供的开发步骤顺利地完成本章节的项目修改。当然，读者如果有任何问题或者想要和笔者讨论的内容，都可以在评论区留下看法，笔者会根据读者的反馈和问题继续整理和完善本章节内容。
