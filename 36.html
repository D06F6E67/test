终于来到了实战环节。相信读者都已经基本掌握Spring Cloud Alibaba微服务相关的组件了，现在就是检验成果的时候了。从本章节开始就要动手搭建最终的微服务架构项目，主要是根据一个单体的商城API项目，并进行微服务化改造，开发过程中将Spring Cloud Alibaba的各个组件像添砖加瓦一样集成到项目里，让newbee-mall-cloud从一个空空的文件夹，慢慢成为一个多模块、多组件、多功能的微服务项目。

编码前的准备

最终的微服务架构实战项目newbee-mall-cloud，是由newbee-mall-api单体项目改造而来的，相关的业务和代码都已经开发完成。后续的实战过程是将单体项目改造为微服务架构项目的一个开发过程。读者在学习后续的实战章节前，一定要先了解和体验newbee-mall-api项目，并参考《最终实战项目的功能介绍》章节中的内容，其中讲解了实战项目的功能点以及改造前的微服务拆分思路，了解这部分内容后对于后续的实战步骤会更有帮助。

后续的实战过程就是笔者根据拆分思路一点一点去完成的，过程中也修改了原单体项目中的一些实现方式和代码。笔者也会介绍开发步骤中遇到的一些小问题，并且把每个步骤的源码都分享给各位读者。虽然笔者对于原来newbee-mall-api单体项目非常熟悉，但是笔者在微服务改造的过程中也遇到了不少问题，毕竟是不同的架构和不同的知识点，很多意料之外的困难都可能出现。只有真正的去做、去编码才会知道有哪些不足，哪些知识点没掌握到，编码的时候遗漏掉了什么。

所以，如果有时间的话，一定要亲自动手改一下这个项目。如果没时间或者觉得暂时还无法独立完成，那就跟着笔者的思路和编码过程来学习和体验。接下来的实战过程都是笔者的开发思路和开发过程，每个重要步骤都会有一份单独的源码，保证读者能够体验到微服务项目开发的每个步骤，体验从0到1的微服务架构项目开发过程。读者也可以根据掌握的知识和相关的项目，自行改造和开发一套微服务架构的项目。

前几个实战章节的讲解会相对详细一些，后续章节则会重点讲解开发思路和过程。因为万事开头难，项目的主体部分一旦完成，后续就会是一些添砖加瓦的重复工作，一切就是水到渠成的事情了。

改造过程中遇到的问题总结

当然，笔者在这个步骤中也是遇到了一个小问题的，在这里直接分享给各位读者，如果在私下自行实战时遇到的话可以直接参考。

在把管理员模块的代码按照对应的目录复制到newbee-mall-cloud-user-service工程后，无法正常启动项目，而是直接报了异常，日志如下：

java
复制代码
org.springframework.context.ApplicationContextException: Failed to start bean 'documentationPluginsBootstrapper'; nested exception is java.lang.NullPointerException

这个问题是由与Spring Boot版本与Swagger版本有些不兼容导致的。当前工程中使用的Spring Boot版本是2.6.3，开发时引入的Swagger版本是2.8.0所以报了异常，后面将Swagger版本改成3.0.0之后，就一切正常了。

当然，Swagger3与Swagger2在整合时的区别是挺大的，依赖项、配置类的写法、默认接口访问地址都有更改，这里需要注意，编码实现时直接参考或者使用笔者提供的源码即可。

总结

万事开头难，这句话对于newbee-mall-cloud实战项目的开发同样适用。凭空要去做一个微服务架构项目的开发，相信对于大部分开发者来说都会有一定的困难，即使笔者刚开始做这个实战项目教程的时候同样有些束手束脚，比如“先做哪一步呢？”、“先引入哪个组件呢？”、“先改造哪个微服务呢？”、“先配置什么东西呢？”等等之类的问题。最终，在思考了一段时间后，开始实际的去编码了。

开发前期肯定要先保证项目结构、依赖关系都配置正常，然后是服务能够正常注册到服务中心、服务接口能够正常调用。当然，这里笔者也加了网关模块，没有选择在开发后期引入服务网关，主要是为了保证服务正常通信，网关模块也能够正常使用。虽然是短短一章内容，但是牵涉到的知识点也是非常充足的，比如Spring Cloud Alibaba依赖项、服务中心Nacos、服务网关Spring Cloud Gateway。

希望各位读者能够根据笔者提供的开发步骤顺利地完成第一个步骤的开发工作。当然，读者如果有任何问题或者想要和笔者讨论的内容，都可以在评论区留下看法，笔者会根据读者的反馈和问题继续整理和完善本章节内容。
